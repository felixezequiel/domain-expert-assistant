# Architecture Decision Records

## Overview

Code without ADR is like law without justification. The next dev will question
the decision, find no answer, and potentially revert something that was chosen
for good reasons. ADR is the project's long-term memory.

**Pre-execution check:** Before starting TDD or any implementation, verify if
the change involves an architectural decision that needs an ADR.

## Flow

1. Check if an ADR already covers the decision (search `docs/adrs/`)
2. If it exists and the decision is compatible, follow the existing ADR
3. If it exists but the decision contradicts it, create an Amendment with date and reason
4. If none exists, create a new ADR before implementing
5. Only after ADR is created/confirmed, proceed to TDD or implementation

## When to Create

| Situation | Needs ADR? | Reason |
|-----------|-----------|--------|
| Framework/library choice | Yes | Impacts entire project |
| Architecture pattern (hexagonal, CQRS, etc.) | Yes | Defines code structure |
| Persistence strategy (SQL vs NoSQL, schema) | Yes | Hard to reverse |
| Service communication pattern (sync vs async) | Yes | Impacts resilience and coupling |
| Auth strategy | Yes | Security, cross-cutting |
| New code convention (naming, structure) | Yes | Affects all devs |
| Test strategy (unit vs integration, libs) | Yes | Defines team workflow |
| Choosing between two valid approaches | Yes | Documents trade-off |
| Punctual bug fix without structural change | No | No architectural decision |
| Refactoring following documented pattern | No | Existing ADR covers it |
| Adding field to existing entity | No | Trivial extension |
| Updating dependency to patch/minor | No | No approach change |

## Templates

### Feature ADR

Required sections:

- **Title** — Format: `ADR-NNN: Short description`
- **Status** — One of: Proposed, Accepted, Deprecated, Superseded by ADR-NNN
- **Date** — YYYY-MM-DD
- **Context** — Current situation motivating the decision. 3-5 objective sentences.
- **Alternatives Considered** — Each with: name, 1-2 sentence description, pros, cons. Minimum 2 alternatives.
- **Decision** — Which alternative was chosen and why. 2-4 sentences.
- **Consequences** — Positive, negative (at least 1), and neutral impacts.
- **Derived Rules** — Optional. Code conventions this decision implies. If code examples needed, link to a knowledge file (e.g., `docs/knowledge/<feature>/<topic>.md`).

### Bug Fix ADR

Use when a bug fix reveals an architectural decision (e.g., change cache strategy, swap algorithm).

Required sections: Title, Status, Date, Bug (incorrect vs expected behavior), Root Cause (architectural flaw, not symptom), Fix Alternatives (include "punctual fix" for comparison), Decision, Consequences.

### Refactoring ADR

Use when refactoring involves pattern/structure/approach change (not cosmetic rename/cleanup).

Required sections: Title, Status, Date, Motivation, Current State, Desired State, Migration Strategy, Risks, Consequences.

## Simplified Mode (Inline)

For smaller decisions: inline comment with prefix `ADR-inline:` followed by what decision was taken, why, and which alternative was discarded. Max 10 lines. If it needs more, create a full ADR.

# Architecture Explainer — On-Demand Deep Dives

## Overview

This skill is a **knowledge index** for the template's architectural patterns.
When the user asks "how does X work", find X in the topics below, give the
concise explanation, point to the actual code, and explain the trade-off.

Always finish with: **(1) where to look in code**, **(2) why it was chosen**,
**(3) what alternative was rejected**.

## Topics

### Hexagonal Layering (Ports & Adapters)

**What:** dependencies point inward. `infrastructure → application → domain`.
Domain has zero framework imports.

**Where:** `src/modules/<context>/{domain,application,infrastructure}/` — strict layering.
`src/shared/` mirrors the same layers for the kernel.

**Why:** lets you swap MikroORM for Prisma, REST for GraphQL, SQLite for Postgres,
without touching domain or application. The dependency direction is enforced by
imports — break it and TS+code review catch it.

**Alternative rejected:** layered/anemic architecture (data classes + service
classes). Fragile because business logic spreads across services and the domain
becomes a record of getters/setters.

### DDD Building Blocks

| Block             | Base class file                                         | Purpose                                                       |
| ----------------- | ------------------------------------------------------- | ------------------------------------------------------------- |
| `Identifier`      | `src/shared/domain/identifiers/Identifier.ts`           | UUID-based identity, subclassed per domain id                  |
| `ValueObject`     | `src/shared/domain/valueObjects/ValueObject.ts`         | Immutable via `Object.defineProperty`, equality by attributes |
| `Entity`          | `src/shared/domain/entities/Entity.ts`                  | Identity + protected props, equality by id                    |
| `AggregateRoot`   | `src/shared/domain/aggregates/AggregateRoot.ts`         | Auto-tracking + `markForDeletion` + `reconstitute`            |
| `DomainEvent`     | `src/shared/domain/events/DomainEvent.ts`               | `{ eventId, eventName, aggregateId, occurredAt, causationId }`|

### Command Factory Pattern

**What:** Commands have a private constructor + static `of(...primitives)` factory.
Adapters call `Command.of(body.name, body.email)` and never construct VOs directly.

**Where:** `src/modules/user/application/commands/CreateUserCommand.ts` is the reference.

**Why:** centralizes VO construction. If `Email` validation changes, only the
factory updates. Adapters stay dumb. Tests assemble Commands the same way HTTP
adapters do — no divergence.

**Alternative rejected:** public constructor receiving VOs. Forces every adapter to
know how to build VOs and duplicates validation logic.

### Automatic Aggregate Tracking

**What:** when an aggregate calls `addDomainEvent(...)`, it auto-registers itself
on a request-scoped tracker. Use cases never call `repository.save()`.

**Where:**

- `src/shared/domain/aggregates/AggregateRoot.ts` — the `setOnTrack` callback hook
- `src/shared/infrastructure/persistence/AggregateTracker.ts` — `AsyncLocalStorage` stack
- `src/shared/infrastructure/persistence/TrackedUnitOfWork.ts` — wires `begin → drain → onCommit`

**How it flows:**
1. App start: `AggregateRoot.setOnTrack(a => AggregateTracker.track(a))`
2. Per-request: `unitOfWork.begin()` pushes a fresh stack frame
3. Use case runs; aggregate's first `addDomainEvent` triggers `track(this)`
4. `commit()` drains tracked aggregates → `onCommit(aggregates)` persists them

**Why:** removes a class of bugs (forgot to call `repository.save()`). Aligns with
"aggregates persist their own consistency boundary" — the dev expresses domain
intent, persistence happens automatically.

**Alternative rejected:** explicit `repository.save(aggregate)` in every use case.
Repetitive, error-prone, and couples use cases to a specific port choice.

### Unit of Work (3 implementations)

**What:** transaction boundary. `begin → commit / rollback`. Same interface, three
implementations swappable by composition root.

**Where:** `src/shared/infrastructure/persistence/adapters/`

| Implementation       | Purpose    | Persistence behavior                                                              |
| -------------------- | ---------- | --------------------------------------------------------------------------------- |
| `MikroOrmUnitOfWork` | Production | Forks EM on begin, routes aggregates to `AggregatePersister`s, flushes inside `em.transactional()` on commit |
| `InMemoryUnitOfWork` | Tests      | Routes aggregates to `InMemoryRepositoryAdapter`s — no DB                          |
| `NoOpUnitOfWork`     | Benchmarks | All operations no-op                                                              |

**Why three:** fast unit tests (no DB), real integration tests (real transaction
semantics), benchmarks free of DB noise.

### ApplicationService — The Orchestrator

**What:** one method, `execute(useCase, command)`. It runs the canonical sequence
for every request:

```
begin → execute → drain → dispatch → publish → saveAll(events) → commit → broadcast
```

**Where:** `src/shared/application/ApplicationService.ts`

**Why one method:** every domain operation goes through the same lifecycle. No
business code re-implements the sequence. EventStore + SSE are wired here as
cross-cutting concerns.

### Event Store (no dual-write)

**What:** every domain event is persisted to the `system_events` table **inside
the same transaction** as aggregate writes. Staged via `em.persist()` before
`unitOfWork.commit()`, so events and state are atomic.

**Where:** `src/shared/ports/EventStorePort.ts` (port) and
`src/shared/infrastructure/persistence/adapters/eventStore/MikroOrmEventStore.ts` (adapter).

**Why:** dual-write (write-then-emit-to-bus) loses events on crash. Same-transaction
write means: event store contains exactly the events that committed. Replay is
trivial.

**Alternative rejected:** outbox pattern. Heavier — requires a poller. Same-transaction
event store gives 90% of the benefit at 10% of the complexity for a monolith.

### SSE (Server-Sent Events) as Cross-Cutting Concern

**What:** after every successful commit, all events are broadcast to two channels:

- `event.aggregateId` — per-aggregate channel (e.g., a UI subscribed to one user's events)
- `__admin__` — global firehose

**Where:**

- `src/shared/ports/SseBroadcasterPort.ts` (port)
- `src/shared/infrastructure/sse/SseBroadcaster.ts` (adapter)
- `src/shared/infrastructure/sse/SseService.ts` (channel-based client manager)
- `src/shared/infrastructure/sse/NoOpSseBroadcaster.ts` (disable SSE in 1 line)

**Why cross-cutting:** modules don't know about SSE. They emit domain events;
broadcast happens at the orchestrator. Adding a new module = automatic SSE.

### Multi-Tenancy (CompanyFilter)

**What:** request-scoped `companyId` via `AsyncLocalStorage`. Tenant-scoped
schemas declare a global filter; `MikroOrmUnitOfWork.onBegin()` reads
`getCurrentCompanyId()` and calls `setFilterParams`. When no tenant is active
(public routes, scheduled jobs), the filter returns `{}` (no-op).

**Where:**

- `src/shared/infrastructure/http/context/TenantContext.ts` — `runWithTenant`, `getCurrentCompanyId`
- `src/shared/infrastructure/persistence/filters/CompanyFilter.ts` — the `companyTenantFilterDefinition`

**Why filter, not where-clause-in-every-query:** filtering at the ORM level means
new queries can't accidentally leak across tenants. Defense in depth.

### MikroORM `PlainObject` Rule (the #1 Footgun)

**What:** every ORM entity class registered in an `EntitySchema` (root + children)
**MUST `extends PlainObject`** from `@mikro-orm/core`.

**Why:** MikroORM uses property-accessor proxies on managed entities. Without
`PlainObject`, `em.upsert(Class, plainInstance)` re-uses identity-mapped proxies
and returns proxy values from getters — mappers see proxies, not primitives.
Hydration breaks in subtle ways.

**Where:** see any entity in `src/modules/user/infrastructure/persistence/mikro-orm/entities/`

### Composition Root + Factories

**What:** `main.ts` is a pure orchestrator. Database setup is `DatabaseFactory.create()`,
shared infrastructure is `InfrastructureFactory.create()`, each module is
`<Context>ModuleFactory.create(emProvider)`.

**Where:**

- `src/shared/factories/` — `DatabaseFactory`, `InfrastructureFactory`
- `src/modules/<context>/factories/` — `<Context>ModuleFactory`

**Adding a new module = 3 lines in main.ts:**

```ts
const fooModule = FooModuleFactory.create(emProvider);
const infrastructure = InfrastructureFactory.create(emProvider, [...userModule.persisters, ...fooModule.persisters]);
fooModule.register(infrastructure, logger);
```

### TypeScript Constraints

- `exactOptionalPropertyTypes: true` — optional props need explicit `| undefined`
- `noUncheckedIndexedAccess: true` — array/record access returns `T | undefined`
- Decorators require SWC (`@swc-node/register/esm-register`) — they don't work with native Node.js type stripping
- Generic constraints: prefer `object` over `Record<string, unknown>` to accept TS interfaces

# Architectural Code Review

## Overview

A diagram is worth more than a list. The reviewer should look at the diagram and
immediately understand WHERE problems are in the execution flow.

## Flow

### Step 1: Analyze the PR

1. Fetch PR details (`gh pr view`) and complete diff (`gh pr diff`)
2. Treat the diff as the **entry point**, not the scope of review
3. Read the full files at PR HEAD (not only diff hunks) to see changes in real context
4. Trace callers and dependents of every changed symbol (functions, classes, types, exports) to assess real impact
5. Identify all findings with severity — including impacts on code **not touched** by the diff but affected by the changes

### Step 2: Map the Execution Cycle

Identify the real runtime flow with the changes applied. Include upstream callers
and downstream dependents that are part of the affected execution path, even when
not in the diff.

For this template, the typical flow is:

```
HTTP/GraphQL adapter
  → Command.of(...)
    → ApplicationService.execute(useCase, command)
      → unitOfWork.begin()
      → useCase.execute(command)
        → aggregate.create() / aggregate method → addDomainEvent()
      → drain tracked aggregates → dispatch / publish events
      → eventStore.saveAll(events)
      → unitOfWork.commit() (transactional flush)
      → sseBroadcaster.broadcastAll(events)
```

### Step 3: Build the Mermaid Diagram

Use `flowchart TD` with subgraphs per architecture layer (HTTP / Application /
Domain / Infrastructure). Solid arrows for main flow, dotted arrows for cross-cutting,

# Domain-Driven Design Patterns

## Overview

DDD provides building blocks for modeling complex business domains. The domain
layer is the core — it contains business rules and has no dependencies on
frameworks or infrastructure.

This template provides ready-to-use base classes for every tactical pattern in
`src/shared/domain/`. Subclass them — don't reinvent.

## Tactical Patterns (in this template)

### Identifier (`shared/domain/identifiers/Identifier.ts`)

UUID-based identity. Subclass once per domain id:

```ts
export class UserId extends Identifier {}
```

Equality by string value. Generate fresh ones with `UserId.generate()` (inherited).

### ValueObject<Props> (`shared/domain/valueObjects/ValueObject.ts`)

**Immutable** via `Object.defineProperty`. Defined only by attributes. No identity.

- Props are `protected` — subclasses expose domain-meaningful getters (`get value()`, `get formatted()`)
- Self-validation in the constructor — throw on invariant violation
- Operations return **new instances** (e.g., `Money.add(other)` returns a new `Money`)

### Entity<Id, Props> (`shared/domain/entities/Entity.ts`)

Object with identity that persists over time. Equality by ID. Methods express
business intent (`approve()` not `setStatusApproved()`).

### AggregateRoot<Id, Props> (`shared/domain/aggregates/AggregateRoot.ts`)

Cluster of entities controlled as one transactional unit. External code references
**by ID only**. One transaction = one aggregate root.

- `addDomainEvent(event)` — auto-tracks the aggregate via `setOnTrack()`
- `markForDeletion()` — routes deletion through the persister on commit
- `static reconstitute(...)` — hydration from persistence WITHOUT firing events

### DomainEvent (`shared/domain/events/DomainEvent.ts`)

Something that **happened**. Name in past tense (`UserCreated`, `AddressAdded`).
Required fields: `eventId`, `eventName`, `aggregateId`, `occurredAt`, `causationId`.

Persisted automatically to `system_events` (event store) **in the same transaction**
as aggregate writes. No dual-write.

### Domain Service

Business logic that **doesn't belong** to any specific entity (operations involving
multiple aggregates or complex calculations requiring external data). Place in
`src/modules/<context>/domain/services/`.

### Repository (Port)

Interface for **persistence** defined in `src/modules/<context>/application/ports/secondary/`
(or `domain/ports/`). Implementation in `infrastructure/persistence/`. One per Aggregate Root.
Returns complete aggregates (already reconstituted).

### Factory

Encapsulates complex creation logic. The aggregate's `create()` static method is
typically enough — explicit factory classes are reserved for multi-step assembly.

## Command Factory Pattern (DRY + Hexagonal)

Commands have a **private constructor** + static `of()` factory that receives
primitives and centralizes VO construction.

```ts
export class CreateUserCommand {
  private constructor(
    public readonly userId: UserId,
    public readonly name: UserName,
    public readonly email: Email,
  ) {}

  public static of(name: string, email: string): CreateUserCommand {
    return new CreateUserCommand(
      UserId.generate(),
      new UserName(name),
      new Email(email),
    );
  }
}
```

Adapters (HTTP, GraphQL) call `CreateUserCommand.of(body.name, body.email)` —
they NEVER construct VOs directly. Format validation in the factory; business
validation in the use case.

## Use Cases — Persistence is Automatic

Use cases **never** call `repository.save()`. The aggregate auto-tracks on the
first `addDomainEvent()`, and `MikroOrmUnitOfWork` flushes everything in a
transaction on commit.

```ts
// ❌ Don't
public async execute(cmd: CreateUserCommand): Promise<User> {
  const user = User.create(cmd.userId, cmd.name, cmd.email);
  await this.repository.save(user);
  return user;
}

// ✅ Do
public async execute(cmd: CreateUserCommand): Promise<User> {
  return User.create(cmd.userId, cmd.name, cmd.email);
}
```

## Strategic Patterns

### Bounded Context

Explicit boundary where one model applies. Each context is a vertical slice in
`src/modules/<context>/` with its own `domain`, `application`, `infrastructure`,
`bootstrap`, and `factories` folders.

### Context Mapping

- **Shared Kernel** — `src/shared/` is the kernel; every module depends on it
- **Customer/Supplier** — express via primary/secondary ports in `application/ports/`
- **Anti-Corruption Layer** — translation adapters in `infrastructure/`
- **Published Language** — domain events (event store gives you a published log for free)

### Ubiquitous Language

Code MUST use business language. `CreditProposal` not `CreditRequest`.
`approve()` not `setStatusApproved()`. Names in tests, ADRs, code, and UI must match.

## Behavior Rules

### MUST
- ALWAYS give entities unique identity and behavior
- ALWAYS make value objects immutable (use `Object.defineProperty` via the base class)
- ALWAYS control consistency through aggregate roots
- ALWAYS define repository interfaces in the application/domain layer (the implementation goes in infrastructure)
- ALWAYS create `of()` factory on every Command
- ALWAYS use ubiquitous language in code
- ALWAYS use `reconstitute()` (not `create()`) when hydrating from persistence

### MUST NOT
- NEVER put business rules outside the domain
- NEVER reference aggregates by object across boundaries (use ID)
- NEVER let adapters build VOs directly
- NEVER create anemic aggregates (data without behavior)
- NEVER call `repository.save()` from a use case
- NEVER use ORM annotations/decorators on domain classes (this template uses `EntitySchema`, no decorators on domain)

# Hexagonal Architecture — Ports & Adapters

## Overview

Dependencies point INWARD (domain). The domain knows nothing about frameworks or I/O.

**Golden rule:** `infrastructure → application → domain`. Domain never imports outward.

## Layers (in this template)

### DOMAIN (`src/modules/<context>/domain/` and `src/shared/domain/`)

Identifiers, ValueObjects, Entities, AggregateRoots, Domain Events, Domain Services,
Repository Ports (interfaces), Domain Exceptions.

**No frameworks.** No `@mikro-orm/*` imports. No HTTP, no SSE, no nodemailer.

### APPLICATION (`src/modules/<context>/application/` and `src/shared/application/`)

Use Cases (orchestration), Commands, Queries, Primary Ports (input interfaces),
Secondary Ports (output interfaces — repository, email, etc.).

**No business rules** (those live in the domain). **No infrastructure imports.**

### PRIMARY ADAPTERS — INPUT (`src/modules/<context>/infrastructure/http/`, `infrastructure/graphql/`)

REST controllers, GraphQL resolvers, message consumers. ALWAYS call `Command.of(...)`.

### SECONDARY ADAPTERS — OUTPUT (`src/modules/<context>/infrastructure/persistence/`, `infrastructure/notifications/`)

Repository implementations, external API clients, message publishers. They IMPLEMENT
ports defined in the inner layers.

### COMPOSITION ROOT (`src/main.ts` + `src/shared/factories/` + `src/modules/<context>/factories/`)

The only place where concrete classes are instantiated and wired. Modules expose a
`<Context>ModuleFactory` that builds persisters and registers routes — `main.ts`
just orchestrates.

## Implementation Direction (Inside-Out)

1. Domain Model — Identifiers, ValueObjects, Entity, AggregateRoot
2. Domain Events
3. Domain Services (if needed)
4. Repository Ports (interfaces)
5. Use Cases (orchestration with `ApplicationService.execute()`)
6. Primary Adapters (REST/GraphQL controllers)
7. Secondary Adapters (MikroORM repository, mappers, persisters)
8. Module bootstrap + factory + `main.ts` wiring

## Folder Layout (vertical slice)

```
src/modules/<context>/
├── domain/
│   ├── identifiers/        # FooId.ts
│   ├── valueObjects/       # FooName.ts, FooStatus.ts
│   ├── aggregates/         # Foo.ts (extends AggregateRoot)
│   └── events/             # FooCreatedEvent.ts
├── application/
│   ├── commands/           # CreateFooCommand.ts (with .of() factory)
│   ├── ports/
│   │   ├── primary/        # input interfaces (rare; usually use cases are the input)
│   │   └── secondary/      # FooRepositoryPort.ts
│   └── usecase/            # CreateFooUseCase.ts (+ .test.ts)
├── infrastructure/
│   ├── http/               # FooController.ts (REST)
│   ├── graphql/            # FooResolver.ts (optional)
│   ├── persistence/
│   │   ├── in-memory/      # InMemoryFooRepository.ts (for tests)
│   │   └── mikro-orm/
│   │       ├── entities/   # FooEntity.ts (extends PlainObject)
│   │       ├── schemas/    # FooSchema.ts (EntitySchema)
│   │       ├── mappers/    # FooMapper.ts (toDomain / toOrmEntity)
│   │       ├── repositories/ # MikroOrmFooRepository.ts
│   │       └── persisters/ # FooAggregatePersister.ts
│   └── notifications/      # (optional) module-specific side effects
├── bootstrap/
│   └── FooModule.ts        # wires deps + registers routes
└── factories/
    └── index.ts            # FooModuleFactory.create(emProvider)
```

## MikroORM ORM Entities — The PlainObject Rule

**All ORM entity classes (every class registered in an `EntitySchema`, including
children) MUST `extends PlainObject` from `@mikro-orm/core`.**

Why: MikroORM uses property accessors (proxies) on managed entities. Without
`PlainObject`, calling `em.upsert(SomeClass, plainInstance)` re-uses identity-mapped
proxies and returns proxy values from getters, breaking mappers and triggering
subtle hydration bugs.

This is the **#1 footgun** of the template. Always check.

## Anti-Patterns

- Domain class with MikroORM imports
- Use case returning an ORM entity
- Controller with business logic
- Command constructor exposed publicly (must be `private`)
- Adapter constructing VOs directly (must call `Command.of()`)
- Use case calling `repository.save()` (the tracker handles it)
- ORM entity not extending `PlainObject`

## Behavior Rules

### MUST
- ALWAYS implement inside-out (domain first, infrastructure last)
- ALWAYS define ports as interfaces in the application/domain layer
- ALWAYS keep domain free of framework dependencies
- ALWAYS use `Command.of()` in primary adapters
- ALWAYS extend `PlainObject` in every MikroORM ORM entity (root + children)
- ALWAYS expose modules through a `<Context>ModuleFactory` that returns `{ persisters, register }`

### MUST NOT
- NEVER put MikroORM/HTTP/GraphQL imports in `domain/` or `application/`
- NEVER let domain depend on infrastructure
- NEVER put business logic in controllers, resolvers, or mappers
- NEVER return ORM entities from use cases (use domain types)
- NEVER instantiate adapters inside use cases (inject via constructor)

# Module Walkthrough — Add Your Domain to the Template

## Overview

Concrete, TDD-first walkthroughs for the four most common "add something" tasks
in this template. Each walkthrough follows **inside-out** order (domain →
application → infrastructure → bootstrap → wiring) and pauses for tests.

Always remind the user: **for a complete copy-paste reference, see
`docs/adding-a-bounded-context.md`** — this skill summarizes the recipe and
adapts to specific asks.

## Walkthrough A — New Bounded Context (full module)

Use when: "I want to add `Product`/`Order`/`Invoice` as a new module."

### Pre-flight

1. **ADR check** — is this an architectural decision worth recording? (e.g., choice of split between two contexts) → activate `skill:adr`
2. **Folder layout:**

```
src/modules/<context>/
├── domain/
├── application/
├── infrastructure/
├── bootstrap/
└── factories/
```

### Step-by-step (TDD inside-out)

1. **Identifier** — `src/modules/<context>/domain/identifiers/<Name>Id.ts` + test
2. **Value Objects** — one file per VO + test (`<Name>Name.ts`, `<Name>Status.ts`, etc.)
3. **Domain Events** — one file per event in `domain/events/` + test (verify `aggregateId` and payload)
4. **Aggregate Root** — `domain/aggregates/<Name>.ts` + test
   - Static `create(...)` emits `<Name>CreatedEvent`
   - Static `reconstitute(...)` for hydration (no events)
   - Domain methods (`approve`, `archive`, etc.) call `addDomainEvent`
5. **Repository Port** — `application/ports/secondary/<Name>RepositoryPort.ts` (interface only)
6. **Command** — `application/commands/Create<Name>Command.ts` with private constructor + static `of(...primitives)`
7. **UseCase** — `application/usecase/Create<Name>UseCase.ts` + test
   - Uses `InMemoryUnitOfWork` and an in-memory repo in tests
   - Wires `AggregateTracker.setOnTrack` in `beforeEach`
   - Returns `Promise<<Name>>` directly (no `.save()` call)
8. **In-Memory Repository** — `infrastructure/persistence/in-memory/InMemory<Name>Repository.ts`
9. **MikroORM ORM Entity** — `infrastructure/persistence/mikro-orm/entities/<Name>Entity.ts`
   - **MUST `extends PlainObject` from `@mikro-orm/core`**
10. **EntitySchema** — `infrastructure/persistence/mikro-orm/schemas/<Name>Schema.ts`
11. **Mapper** — `infrastructure/persistence/mikro-orm/mappers/<Name>Mapper.ts` (`toDomain` / `toOrmEntity`)
12. **Aggregate Persister** — instantiate `MikroOrmAggregatePersister` with a config (no need for a class):

    ```ts
    new MikroOrmAggregatePersister({
      aggregateClass: Foo,
      ormEntityClass: FooEntity,
      toOrmEntity: FooMapper.toOrmEntity,
      cleanupNestedEntities: () => [],   // optional
      deleteEntity: (em, foo) => em.removeAndFlush(...),  // optional
    })
    ```

13. **MikroORM Repository** — `infrastructure/persistence/mikro-orm/repositories/MikroOrm<Name>Repository.ts` (uses `EntityManagerProvider` + mapper)
14. **HTTP Controller** — `infrastructure/http/<Name>Controller.ts`. Calls `Command.of(...)` and `applicationService.execute(useCase, command)`. Maps domain → DTO for response.
15. **Module Bootstrap** — `bootstrap/<Name>Module.ts` (registers routes on `httpServer`)
16. **Module Factory** — `factories/index.ts` exporting `<Name>ModuleFactory.create(emProvider)` returning `{ persisters, register }`
17. **Wire in `main.ts`:**

    ```ts
    const fooModule = FooModuleFactory.create(emProvider);
    const infrastructure = InfrastructureFactory.create(emProvider, [
      ...userModule.persisters,
      ...fooModule.persisters,
    ]);
    fooModule.register(infrastructure, logger);
    ```

18. **Migration** — `npm run migration:create -- --name=Create<Name>Table`
19. **Smoke test** — `npm start` and `curl` your new endpoint
20. **Doc sync** — update `README.md` API surface table; if you made a decision worth recording, add an ADR; activate `skill:doc-sync`

### What to delegate to TDD

Steps 1, 2, 3, 4, 7 are **TDD steps**. For each: write the test first, run it (RED),
implement minimum code (GREEN), then move on.

## Walkthrough B — New Aggregate inside an existing module

Use when: "Add a `Subscription` aggregate to the `User` module."

1. New `domain/identifiers/<Name>Id.ts` + test
2. New `domain/aggregates/<Name>.ts` + test (with `create` and `reconstitute`)
3. New `domain/events/` files + tests
4. New `application/ports/secondary/<Name>RepositoryPort.ts`
5. New `application/usecase/Create<Name>UseCase.ts` + test
6. New `infrastructure/persistence/{in-memory,mikro-orm}/...` (entity, schema, mapper, persister, repo)
7. **Add the new persister to the module's `factories/index.ts` `persisters` array**
8. Register HTTP/GraphQL routes in `bootstrap/<Module>Module.ts`
9. Migration + smoke test + doc sync

## Walkthrough C — New Use Case in an existing module

Use when: "Add a `BlockUserUseCase` to the User module."

1. **Domain method** — if behavior doesn't exist yet, add it on the aggregate (e.g., `User.block()` emitting `UserBlockedEvent`) — TDD with aggregate test first
2. **Event** — new `domain/events/UserBlockedEvent.ts` + test
3. **Command** — `application/commands/BlockUserCommand.ts` with `.of(userId)`
4. **UseCase** — `application/usecase/BlockUserUseCase.ts` + test using `InMemoryUnitOfWork`
5. **Adapter** — add a route in `infrastructure/http/UserController.ts` (e.g., `POST /users/:userId/block`)
6. **Module bootstrap** — register the new route in `UserModule.ts`
7. **Doc sync** — update `README.md` API surface, run `skill:doc-sync`

No new persister. No new factory. The existing aggregate already has its persister.

## Walkthrough D — New Domain Event for an existing aggregate

Use when: "When user changes email, emit `UserEmailChangedEvent`."

1. **Event class** — `domain/events/<Name>Event.ts` + test
   - Constructor receives `aggregateId` and the payload primitives
   - Test asserts the payload shape
2. **Aggregate method** — add or modify the method that emits the event:

   ```ts
   public changeEmail(newEmail: Email): void {
     // invariant checks...
     this.props.email = newEmail;
     this.addDomainEvent(new UserEmailChangedEvent(this.id.toValue(), newEmail.value));
   }
   ```

3. **Aggregate test** — assert the new event is emitted on the right call
4. **Optional: domain event handler** — if a side effect is needed (e.g., send email), add it in `application/usecase/` and register on `DomainEventManager` in the module bootstrap
5. **Event store + SSE** — already automatic. The new event will be persisted in `system_events` and broadcast to its `aggregateId` channel and `__admin__`. No infrastructure work needed.
6. **Doc sync** — if event semantics matter, mention it in the module's `bootstrap/<Module>Module.ts` registration comments

## Common Pitfalls (call these out proactively)

- **Forgot `extends PlainObject` on the new ORM entity** → MikroORM hydration bugs
- **Use case calling `repository.save()`** → unnecessary; remove
- **Adapter constructing VOs** → use `Command.of(...primitives)`
- **New aggregate but persister not added to `persisters` array** → won't persist on commit
- **Public constructor on Command** → must be `private`
- **Missing `reconstitute()` static** → repository hydrations will emit events on every read

## Behavior Rules

### MUST
- ALWAYS follow inside-out order (domain → application → infrastructure)
- ALWAYS write the test before the production code (TDD)
- ALWAYS pause to confirm RED before writing GREEN
- ALWAYS remind the user about the `PlainObject` rule when creating ORM entities
- ALWAYS update `README.md` API surface and run doc-sync at the end
- ALWAYS check for ADR-worthy decisions before starting (handoff to `skill:adr`)

### MUST NOT
- NEVER skip the test step — even for "trivial" steps like Identifiers
- NEVER place ORM imports in `domain/`
- NEVER call `repository.save()` from a use case
- NEVER add a new persister without including it in `<Context>ModuleFactory.persisters`
- NEVER create a Command with a public constructor

# Readable Code

## Overview

Code must be readable like text. A dev who has never seen the language should be
able to understand the business logic reading top to bottom, without researching
language-specific syntax.

## Rules

### 1. Zero Magic Numbers

Every literal value must be extracted to a named constant describing its purpose.

**Accepted exceptions:**
- `0`, `1` as array indices (`list[0]`)
- HTTP status codes when self-evident in context (200, 401, 404, 500)
- Year literals (1900-2099)
- Loop counters in iteration ranges

### 2. No Syntax Sugar

Prioritize explicit forms any dev can read without advanced language knowledge.

**Functional chains:** prefer `for...of` loops over chained `.map().filter().reduce()` when the chain has 3+ steps or contains business logic. One-step transformations are fine.

**Nested ternaries:** forbidden. Use explicit `if/else` blocks.

**Optional chaining abuse:** prefer explicit `if (x && x.y && x.y.z)` when it makes intent clearer than `x?.y?.z`.

**Destructuring with rename:** avoid `const { a: b } = obj` — prefer `const b = obj.a`.

**Expression-body arrow functions:** accepted ONLY for simple delegation, factory methods, or single short expressions (1-3 lines, no intermediate variables needed). Complex logic MUST use block body with named intermediate variables.

### 3. Declarative Variables (Step by Step)

Every compound expression must be decomposed into named intermediate variables.
The dev should not need to interpret the expression to understand what it does.

```ts
// Bad
return items.filter(i => i.qty > 0 && i.expiresAt > Date.now()).map(i => i.id);

// Good
const validIds: string[] = [];
for (const item of items) {
  const hasStock = item.qty > 0;
  const notExpired = item.expiresAt > Date.now();
  if (hasStock && notExpired) {
    validIds.push(item.id);
  }
}
return validIds;
```

### 4. Descriptive Names

Variables must describe what they represent — no generic abbreviations like `tmp`, `x`, `data`, `res`, `obj`.

### 5. Complex Algorithm Documentation

Non-trivial algorithms must have a JSDoc comment explaining what they do, why they exist,
and how they work. Default to no comment — only add when the WHY is non-obvious.

## DDD-Specific Guidance

- **Value Object literals** (max length, min length, regex) → `private static readonly` constants on the VO class.
- **Magic strings** (event names, status enums, role names) → import from constants/enums, never inline strings.
- **Test setup** repeated → use a `makeFoo()` helper, not duplicated literal objects.

## Behavior Rules

### MUST
- ALWAYS extract literal numbers to named constants
- ALWAYS prefer explicit `for...of` over functional chains with 3+ steps or business logic
- ALWAYS decompose compound expressions into named intermediate variables
- ALWAYS use full descriptive names (no abbreviations)
- ALWAYS document complex algorithms with JSDoc

### MUST NOT
- NEVER leave literal numbers without a named constant (except indices and trivial cases)
- NEVER use nested ternaries
- NEVER chain `.map().filter().reduce()` when it hides business logic
- NEVER use single-letter variable names except for loop counters
- NEVER assume the dev knows TypeScript-advanced patterns

# Safe Refactoring — Feature Toggle Strategy

## Overview

Refactoring NEVER modifies existing code directly. Always create a parallel
implementation with a feature toggle for instant rollback.

## Mandatory Flow

### Phase 1: Test Preparation

1. Identify existing tests covering the behavior
2. Duplicate tests for the new implementation
3. Increase coverage if gaps are found
4. Fix failing tests before proceeding
5. Green baseline: all tests must pass

### Phase 2: Parallel Implementation

1. Create new implementation in a separate location (e.g., `FooV2.ts` next to `Foo.ts`)
2. Keep old code 100% intact — do not modify, do not delete
3. Both implementations coexist

### Phase 3: Feature Toggle

1. Create feature toggle to switch between implementations
2. Toggle must be externally configurable (env var, config flag) — not hardcoded
3. Default: old implementation (safety)

### Phase 4: Deploy and Validation

1. Deploy with toggle disabled (uses old implementation)
2. Enable gradually: staging first, canary release, full rollout
3. Monitor metrics, errors, and correlation IDs
4. If problems: disable toggle = instant rollback

### Phase 5: Cleanup (After Stabilization)

Only after a stability period in production:

1. Remove old implementation
2. Remove feature toggle
3. Consolidate tests (remove duplication)
4. Update documentation, ADRs (status: Superseded), README

## Composition Root Toggling

In this template, the composition root is `src/main.ts`. Wire toggles there:

```ts
const useNewPersister = process.env.USE_NEW_PERSISTER === "true";
const persister = useNewPersister
  ? new NewUserAggregatePersisterV2(...)
  : new UserAggregatePersister(...);
```

This keeps the toggle at the outermost layer; modules don't know about it.

- ALWAYS ask about existing tests before refactoring
- ALWAYS create parallel implementation
- ALWAYS suggest feature toggle
- ALWAYS keep old code intact until stabilization

### MUST NOT
- NEVER modify existing code directly
- NEVER delete old code before production validation
- NEVER refactor without feature toggle
- NEVER assume refactor is "done" before validation

# Task Breakdown — Surgical ADR Decomposition

## Overview

Each PR must be the smallest unit of work that adds value to the product. Large
PRs are risky PRs. Surgical PRs pass review in minutes, avoid conflicts, and
deliver exactly what they promise.

## Decomposition Priorities (in order)

### 1. Add Value with the PR

Each task must deliver something functional or structural when completed.

**Test:** if the PR were merged and nothing else done, would the project be better?

### 2. Minimum Cognitive Load

Each PR should require minimal context for review.

**File count is NOT a complexity criterion.** Hexagonal architectures generate
many files by nature. 15 files of predictable boilerplate are simpler than 3
files of complex logic.

**What defines complexity:**

- Number of independent concepts the reviewer must understand simultaneously
- Number of non-obvious decisions (business logic, algorithms, trade-offs)
- Whether the PR crosses module boundaries
- Amount of existing behavior altered (vs new code that breaks nothing)

### 3. Full Coverage

Sum of all tasks must cover 100% of the scope defined in the ADRs.

### 4. Dependency Order

Tasks must respect the natural dependency tree:

- Types/interfaces before implementations
- Domain before Application before Infrastructure (inside-out)
- Shared/base before feature-specific

**Rule:** a task can only depend on tasks with a lower number.

### 5. Distribution with Minimum Conflicts

When multiple devs: divide by natural code boundaries.

**Golden rule:** two devs should never need to edit the same file in the same sprint.

## Mandatory Flow

1. **Inventory ADRs** — list all relevant ADRs in `docs/adrs/`, map dependencies, read knowledge files
2. **Extract Requirements** — atomic implementable requirements from each ADR
3. **Group into Tasks** — tasks that can be a single PR, respect complexity limits
4. **Order by Dependency** — technical, ADR, and layer dependencies (inside-out)
5. **Classify Complexity** — evaluate by 4 cognitive-load axes
6. **Assign** (if multiple devs) — group by natural boundaries (typically by module)

## Complexity Classification

4 axes: simultaneous concepts, non-obvious decisions, crossed boundaries, altered behavior.

| Axis                  | Low                  | Medium                       | High                              |
| --------------------- | -------------------- | ---------------------------- | --------------------------------- |
| Simultaneous concepts | 1-2                  | 3-4                          | 5+                                |
| Non-obvious decisions | 0-1                  | 2-3                          | 4+                                |
| Crossed boundaries    | 1 module             | 2 touching modules           | 3+ or distant modules             |
| Altered behavior      | New code only        | Alters 1-2 existing flows    | Alters 3+ or critical flow        |

**High complexity is FORBIDDEN** — must be broken into smaller subtasks.

## Task Numbering

Format: `T-XX.YY` (module.sequential). With tracks: `T-XX.A01` (dev A), `T-XX.B01` (dev B).

## Decomposition Strategies for This Template

Given the template's vertical-slice module pattern, the most natural breakdown for
a new bounded context is **by layer**:

1. Domain (IDs, VOs, AggregateRoot, Events) — one PR
2. Application (Commands, Ports, UseCases) — one PR
3. Infrastructure persistence (in-memory + MikroORM persister + mapper) — one PR
4. Infrastructure adapters (HTTP/GraphQL controllers) — one PR
5. Bootstrap + factory + `main.ts` wiring + migration — one PR

Or **by subfeature** (vertical slice through all layers) when each subfeature is small.

## High Complexity Breaking Techniques

1. Separate Interface vs Implementation
2. Separate by Operation (split CRUD into individual endpoints)
3. Separate Infrastructure from Logic
4. Separate Base from Feature

# TDD Workflow — Red Green Refactor

## Overview

Every implementation follows the mandatory TDD cycle: Red -> Green -> Refactor.
Tests are ALWAYS written before production code.

## Feature Flow

```
RED      -> Write test first (MUST fail)
GREEN    -> Implement MINIMUM code to pass
REFACTOR -> Improve while keeping tests green
```

### Implementation Steps

1. Identify the behavior to implement
2. Create test file in the correct location (co-located: `Foo.test.ts` next to `Foo.ts`)
3. Write the test describing expected behavior
4. Run the test — MUST FAIL (RED)
5. Verbally confirm the test fails and why
6. Implement minimum code for the test to pass
7. Run the test — MUST PASS (GREEN)
8. Ask the user if they want to refactor

## Bug Fix Mode

1. Write test that reproduces exactly the bug (MUST fail)
2. Confirm the test fails (reproduces the bug)
3. Fix with minimum code
4. Confirm the test passes
5. The test serves as documentation and regression prevention

## Test Structure (Co-Located)

This template uses co-located tests — each `Foo.ts` has its `Foo.test.ts` next to it:

```
src/modules/user/domain/User.ts
src/modules/user/domain/User.test.ts

src/modules/user/application/usecase/CreateUserUseCase.ts
src/modules/user/application/usecase/CreateUserUseCase.test.ts
```

Integration tests live in `<module>/integrationTests/`. Performance benchmarks in `<module>/performanceTests/`.

## Implementation Order (Inside-Out)

For hexagonal architecture, follow inside-out test order:

1. **Domain model tests** — IDs, ValueObjects, Entities, AggregateRoots
2. **Domain event tests** — verify event payload and `aggregateId`
3. **Use case tests** — with `InMemoryUnitOfWork` and in-memory repositories
4. **Adapter tests** — HTTP controllers, GraphQL resolvers, mappers
5. **Integration tests** — real `MikroOrmUnitOfWork` with SQLite

```ts
  });

    // arrange InMemoryUnitOfWork + repository
    // act through ApplicationService
    // assert aggregate state + emitted events
  });
});
```
